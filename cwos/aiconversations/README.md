# aiconversations/ templates

This folder ships **template files** for the two centralized derived views that CWOS-aligned repos commonly maintain at the root of their own `/aiconversations/` directory:

- **`0-work-status.template.md`** — Work-status dashboard. Answers *"which threads need attention?"* (conversation-level activity, sorted by most-recent within each table). Customize the table set for your repo's content domains.
- **`0-project.template.md`** — Open-projects rollup. Answers *"what specific projects are open?"* (project-level rollup walked from per-conversation `-tasks.md` files). Regenerated by the `open-projects` skill once instantiated.

Both files use `{{DATE}}` and other placeholders that get replaced when copied into a target repo. Drop the `.template` suffix when copying.

## Canonical structure rules (apply to both files in every CWOS-aligned repo)

These are the structural invariants. Per-repo content (specific table labels, project lists, activity rows) varies; the structure does not.

### Frontmatter schema

Both `0-work-status.md` and `0-project.md` use a parallel lean YAML frontmatter:

```yaml
---
documentType: <Work-Status Dashboard | Open Projects Rollup>
purpose: <one-line description>
version: <semver, optional but recommended>
versionNote: <one-line, optional — document recent refreshes here>
lastUpdated: <YYYY-MM-DD>
lastUpdatedNote: <multi-line, optional — for activity-since-last-refresh detail; useful in 0-work-status.md>
status: <Active - In Use | Active - Stub | Archived>
location: /aiconversations/<filename>
companionSurface: /aiconversations/<the other file>
generatedBy: /operations/cwos/skills/open-projects/SKILL.md   # 0-project.md only
---
```

Drop unused optional fields rather than carrying empty ones. Keep `companionSurface:` pointing at the sibling file in both directions.

### Section structure

**`0-work-status.md`:**

1. Brief intro paragraph + refresh-invocation note + canonical-structure note
2. **N tables** as `## 1. Domain Name` through `## N. Domain Name` (3-5 typical). Each table has columns: `# | Conversation | Folder | Size | Last Activity Summary`. Rows sorted by most-recent activity within each table.
3. **Maintenance notes** (or "Notes") section after the tables — refresh discipline, recent maintenance, structural conventions.
4. **Conversational Work Help** section — the canonical skill-invocation reference block (same content across all CWOS repos for cross-repo consistency; see existing implementations in chevan-content / workspace-chevan / this template).

**`0-project.md`:**

1. Brief intro paragraph + refresh + prioritize invocation notes + canonical-structure note
2. **N domain sections** as `## 1. Domain Name` through `## N. Domain Name` — numbering and labels MUST mirror `0-work-status.md`'s table set. Each section has nested `### [tasks-file-name.md](path)` H3 entries; under each H3, a bullet list of open projects (ideally with `**Project N:**` prefix from the source file).
3. **Files needing cleanup** section (optional) — `-tasks.md` files that don't yet follow the standard layout; surfaced by the `open-projects` skill.
4. **Notes on this rollup** section — regeneration policy, format expectations, link to the tasks-file standard in AICONFIG.md.

### Mirror rule

The single most important canonical pattern: `0-project.md` section numbering and labels MUST mirror `0-work-status.md` table set. This makes cross-reference between the two surfaces explicit — Table 2 in the dashboard corresponds to Section 2 in the rollup. Operators (and the `prioritize-open-projects` skill) rely on this parity.

### Tasks file structure

See `AICONFIG.template.md` → `## STANDARDS - OPERATIONS` → `### Project tracking` → `#### Tasks File Structure (Standard)` for the canonical four-section layout (Summary of Open Projects / Notes/Reference / Open Projects / Closed / Completed Projects), four-level hierarchy (`### Project N:` / `#### Phase N:` / `**Deliverable**` / `- [ ] Task`), and rationale.

## Bootstrap procedure

When setting up a new CWOS-aligned repo with the `aiconversations/` extension:

1. Copy `0-work-status.template.md` to `<target>/aiconversations/0-work-status.md` (drop the `.template` suffix). Replace `{{DATE}}` placeholders. Replace the `{{TABLE N NAME}}` placeholders with table groupings that match your repo's content domains (life-domain, business-domain, content-domain, etc.).
2. Copy `0-project.template.md` to `<target>/aiconversations/0-project.md` (drop the `.template` suffix). Replace `{{DATE}}` and `{{DOMAIN N}}` placeholders.
3. Author the repo-specific `work-status-dashboard` skill at `operations/cwos/skills/work-status-dashboard/SKILL.md`. The CWOS starter does NOT ship a generic `work-status-dashboard` skill because the table set is repo-specific and a generic skeleton would be more misleading than helpful. Use the existing `open-projects/SKILL.md` as a structural reference for skill format.
4. Invoke `open-projects` to do the first real population of `0-project.md` from this repo's actual `-tasks.md` files.

## When to refresh

- **`refresh-work-management`** — orchestrator; refreshes both files. Common case.
- **`work-status-dashboard`** alone — when only conversation-level activity changed.
- **`open-projects`** alone — when only `-tasks.md` files changed.
- **`prioritize-open-projects`** — read-only analysis of the rollup; does NOT refresh the rollup. Run `refresh-work-management` first if the rollup is stale.

## Why the dashboard is here (and the dashboard-refresh skill isn't)

The dashboard FILE follows a stable shape (frontmatter + tables-by-domain), so a template version is useful — bootstrappers can copy it and customize the table set without re-deriving the structure.

The dashboard-refresh SKILL is intentionally repo-specific. Different repos need different sort orders, different size-flagging thresholds, and different commit-history queries. Shipping a generic skeleton would push complexity onto every bootstrapper to immediately customize it. Better to leave it for the bootstrapper to author once they understand their own content shape.

## Why `0-project.md` ships a template

`0-project.md` is *generated*, not *authored*. The template exists so the first run of `open-projects` has something to write to (and the bootstrapper has a structural reference for what the file becomes). After first generation, the template's placeholder content is replaced with real data and the file is maintained by the skill.
